# RFC Action File Format

The output of Step 7. A single markdown file written to the project root. Do not
open GitHub issues or Jira tickets directly — write the file and hand off.

---

## File naming

```
arch-rfcs-YYYY-MM-DD.md
```

```markdown
# Arch RFC Batch — YYYY-MM-DD

> **Repo:** owner/repo-name
> **Jira project:** PROJ
> **Generated by:** arch-lens skill
> **Scope:** path/analysed (or "full repository")

---
```

Omit repo or Jira project lines if not provided. Ask the user for these at the
start of Step 7 if they are not already known from context.

then cognitive load, then interface stability). Each section maps directly to a
GitHub issue or Jira ticket body — no further interpretation needed.

**Dependency category:** In-process | Local-substitutable | Remote but owned | True external

---

## Priority and effort mapping

| Impact score | Priority |
|---|---|
| Blocks testability of a core module | Critical |
| High fan-in module with coupling issue | High |
| Medium scope, isolated domain | Medium |
| Leaf module, low blast radius | Low |

| Effort | Signals |
|---|---|
| XS | Rename, move, or inline — under 1 hour |
| S | Interface change, update callers — half day |
| M | New interface + adapter + tests — 1–2 days |
| L | Boundary refactor across multiple modules — 3–5 days |
| XL | Cross-team coordination, phased migration — over 1 week |

| Remote but owned | `ports-and-adapters` |

## RFC-001: Deepen PaymentProcessor — hide retry and idempotency behind process()

**Priority:** High
**Effort:** M
**Labels:** arch-rfc, payments, ports-and-adapters
**Dependency category:** Remote but owned (Ports & Adapters)

---

### Problem

`PaymentProcessor` is a shallow module: its public interface exposes `retryCount`,
`idempotencyKey`, and `timeoutMs` as caller responsibilities, pushing infrastructure
orchestration into every call site. Seven call sites each re-implement the same
retry sequence (`src/orders/OrderService.ts:89`, `src/webhooks/WebhookHandler.ts:44`,
and five others). The seam between payment logic and retry infrastructure is where
bugs accumulate — and it belongs to no one.

### Proposed Interface

**Signature:**
```ts
interface PaymentProcessor {
  process(payment: Payment): Promise<Receipt>
}
```

**Usage example:**
```ts
const receipt = await processor.process(payment)
```

**What complexity is hidden:**

- Retry count, backoff policy, and jitter — owned by the implementation
- Idempotency key generation and deduplication storage
- Timeout configuration sourced from environment at construction time
- `shouldRetry()` classification logic removed from all call sites

### Dependency Strategy

**Remote but owned (Ports & Adapters).** Define port `RetryPolicy` and port
`IdempotencyStore`. Production adapter: `HttpRetryPolicy` (real backoff over HTTP).
Test adapter: `InMemoryIdempotencyStore` + `FixedRetryPolicy` (deterministic, no network).

### Testing Strategy

**New boundary tests to write:**
- `process(payment)` succeeds on first attempt → returns `Receipt`
- `process(payment)` fails twice, succeeds on third → `InMemoryIdempotencyStore` records one entry
- `process(payment)` exhausts retries → throws `PaymentFailedError` with original cause
- Duplicate `process(payment)` call with same payment → idempotency store short-circuits, no second charge

**Old tests to delete:**
- `PaymentProcessorTest.test_retry_on_network_failure` — mocks `HttpClient` internally; replaced by boundary retry test above
- `PaymentProcessorTest.test_idempotency_key_generation` — tests internal key format; replaced by duplicate-call boundary test above
- `OrderService.test_payment_retry_logic` — retry logic no longer belongs to `OrderService`

**Test environment needs:** None — `InMemoryIdempotencyStore` and `FixedRetryPolicy` run in-process.

### Implementation Recommendations

- **What the module should own:** retry policy execution, idempotency key lifecycle, timeout enforcement
- **What it should hide:** key format, retry algorithm, HTTP adapter details, timeout source
- **What it should expose:** `process(payment: Payment): Promise<Receipt>` — one entry point, one concept
- **How callers should migrate:** mechanical — remove `retryCount`, `idempotencyKey`, `timeoutMs` from all seven call sites; replace with single `process(payment)` call. No coordination required — all callers are in the same repo.

---
```
