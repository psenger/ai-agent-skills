# File Structure

The audit can land on **three structurally distinct targets**. Each has its own schema. Findings valid for one target are often structurally impossible (and therefore wrong) for another. Always identify the target first.

## Quick detection table

| Signal in cwd | Target |
|---|---|
| `agent-os/standards/index.yml` exists | **B** — project install |
| One `standards/` dir or one set of domain folders at cwd root, no sibling profile dirs | **A** — single profile source |
| Multiple sibling profile-shaped dirs (with or without a `profiles/` wrapper), no `index.yml` | **C** — enterprise profiles repo |

When in doubt, ask the user. Open every findings report with the detected target.

---

## Target A — single profile source directory

A profile as it lives in `~/agent-os/profiles/<name>/` (or anywhere a single profile is checked out for editing). The base installation builds the project install from one or more of these via `project-install.sh`.

**Two valid layouts exist in the wild.** Both are accepted by the audit.

### Layout A1 — `standards/` wrapper (canonical, install-script-resolvable)

```
<profile-name>/
  standards/
    <domain>/
      <standard>.md
    <root-level-standard>.md
```

This is what `project-install.sh` resolves: it reads from `<profile>/standards/` when building the merged project install. New profiles created via `~/agent-os/scripts/sync-to-profile.sh --new-profile <name>` produce this layout.

### Layout A2 — domain folders at profile root (no `standards/` wrapper)

```
<profile-name>/
  <domain>/
    <standard>.md
  <root-level-standard>.md
```

This is the layout of the upstream `default` profile shipped in the Agent OS repo (e.g. `profiles/default/global/tech-stack.md`). Older custom profiles also appear in this shape.

### What is true for both layouts

- **No `index.yml`.** The index is generated at project-install time, not hand-authored in a profile source. An `index.yml` present in a profile source dir is a finding (likely a copy-paste from a project install).
- **No `agents/`, `commands/`, `workflows/`, or `profile-config.yml`.** Those are v2 artifacts.
- **Standards files are `.md` only.**
- **Profile source dirs do not contain `specs/` or `product/`.** Those belong to project installs.

### Audit notes per layout

- If the audit detects Layout A2 and the user intends to install this profile via `project-install.sh`, **flag a warning**: the script resolves `<profile>/standards/`, so a Layout-A2 profile installs nothing. The fix is to move domain folders under a `standards/` subdir, or to handle the install manually.
- If the audit detects Layout A2 in the upstream `default` profile, note it as a known upstream anomaly, not user error.

## Target B — project installed standards

A repository that has been installed into via `~/agent-os/scripts/project-install.sh`. This is the most common audit target.

```
<repo>/
  agent-os/
    standards/
      <domain>/
        <standard>.md
      <root-level-standard>.md
      index.yml                 # GENERATED by project-install.sh — not hand-authored
    specs/
      YYYY-MM-DD-HHMM-<slug>/
        plan.md
        shape.md
        standards.md
        references.md
        visuals/
    product/
      mission.md
      roadmap.md
      tech-stack.md
  .claude/
    commands/
      agent-os/                 # 5 slash command .md files (copied from base)
        discover-standards.md
        index-standards.md
        inject-standards.md
        plan-product.md
        shape-spec.md
```

### `agent-os/standards/` is a merged artifact

Target B's `standards/` directory is **not a copy of any single profile**. It is the merged output of an inheritance chain. If `config.yml` declares `my-app -> rails-base -> ruby-general`, then `agent-os/standards/` contains files from all three profiles, flattened by domain. Child files override parent files of the same name.

This means:
- A domain like `api/` may contain files contributed by multiple profiles in the chain. There is no profile-name nesting at this layer.
- Two unrelated-looking domain folders (`symfony/` and `php/`) sitting side by side is **expected** if the chain merged contributions from different profiles. This is not a finding.
- The `index.yml` here is generated by `create_index()` in `project-install.sh`. Its presence is **required** in Target B; its absence is a blocking finding. Do not flag it as "shouldn't be hand-authored" — it isn't.

### `.claude/commands/agent-os/` must contain exactly 5 files

`discover-standards.md`, `index-standards.md`, `inject-standards.md`, `plan-product.md`, `shape-spec.md`. Missing files mean a stale or partial install; the fix is `~/agent-os/scripts/project-install.sh --commands-only`.

### What to commit

| Path | Commit? | Reason |
|---|---|---|
| `agent-os/standards/` | Yes | Share conventions with the team |
| `agent-os/product/` | Yes | Share product context |
| `agent-os/specs/` | Yes | Preserve project history |
| `.claude/commands/agent-os/` | Optional | Team can regenerate via `--commands-only` |

## Target C — enterprise profiles repository

A standalone git repo containing multiple named profiles, typically cloned into `~/agent-os/profiles/` (or used as a versioned drop-in for the whole base install). Used by organizations to share profiles across engineers (pattern 3 in `profiles.md`).

**Two valid layouts exist.** The audit auto-detects which one applies.

### Layout C1 — `profiles/` wrapper at repo root

```
<repo-root>/
  profiles/
    rails/
      standards/...
    nextjs/
      standards/...
    client-acme/
      standards/...
  README.md
```

This shape lets the repo be cloned in place of `~/agent-os/` itself, or its `profiles/` subdir replaces `~/agent-os/profiles/`. Often paired with a `config.yml` at repo root that declares inheritance for the bundled profiles.

### Layout C2 — profile dirs at repo root, no wrapper

```
<repo-root>/
  rails/
    standards/...
  nextjs/
    standards/...
  client-acme/
    standards/...
  README.md
```

Contents are cloned/copied directly into `~/agent-os/profiles/`. No `profiles/` wrapper; the repo root *is* the profiles parent.

### What is true for both layouts

- **No `index.yml` anywhere in the repo.** Each profile is a source; indexes are generated at project-install time. An `index.yml` in a Target C repo is a finding.
- **Multiple named profile dirs.** A single-profile repo would be Target A, not Target C.
- **Each profile dir follows Target A schema** (Layout A1 or A2). The Target A rules apply per-profile.
- **No `agents/`, `commands/`, `workflows/`, or `profile-config.yml`** in any profile. v2 artifacts.
- **Has its own install mechanism.** `project-install.sh` does not directly consume a Target C repo. Installation is typically: `git clone <repo> ~/agent-os/profiles/` (Layout C2) or copy the contents of `profiles/` into `~/agent-os/profiles/` (Layout C1).
- A `config.yml` at repo root, if present, declares inheritance among the bundled profiles. It mirrors `~/agent-os/config.yml` and follows the same schema.

### Findings that do NOT apply to Target C

- Missing `index.yml` (it does not belong here).
- Missing `.claude/commands/agent-os/` (Target C is not a project install).
- Missing `agent-os/product/` or `agent-os/specs/` (those belong to Target B).

## Base installation `~/agent-os/`

Not an audit target on its own, but referenced when validating `config.yml` during Target A or Target C audits.

```
~/agent-os/
  commands/
    agent-os/                   # 5 slash command .md source files
  profiles/
    <profile-name>/             # each follows Target A schema
  scripts/
    project-install.sh
    sync-to-profile.sh
    common-functions.sh
  config.yml                    # version, default_profile, profiles: (inheritance)
```

---

## Things that should NOT exist in any target (v2 leftovers)

Flag on sight in any target:

- `.claude/agents/agent-os/` (subagents, retired in v3)
- `profile-config.yml` anywhere under a profile dir (inheritance moved to `~/agent-os/config.yml`)
- `agents/`, `commands/`, or `workflows/` subdirs inside a profile dir
- `~/agent-os/config.yml` missing, or `version` below `3.0.0`
